# TrueUsed 需求文档初稿

版本：2026-07-06

## 1. 产品目标

TrueUsed 做一个“可信二手交易”MVP。核心不是做大而全商城，而是跑通发布、购买、支付、发货、验货报告、售后退款这条闭环。

Git 仓库：[xshsama/True_Used.git](https://github.com/xshsama/True_Used.git)

目标用户：

- 买家：想降低买二手的踩坑风险。
- 卖家：想快速发布、成交、收款。
- 平台管理员：处理用户、提现、验货和售后问题。

买家信任不足的解决方式：

- 商品信息透明：统一展示价格、成色、图片、卖家、交易模式。
- 平台验货背书：平台验货商品生成验货结果和报告，并绑定到订单页。
- 过程可追踪：支付、发货、物流、确认收货、售后状态都可查看。
- 售后兜底：买家可申请退款，卖家处理后形成完整记录。

本期不做：真实物流接入、真实线下验货工单、复杂风控、AI 自动决策、多商户清结算。

## 2. 功能范围

| 模块 | 本期能力 | 边界 |
| --- | --- | --- |
| 商品 | 发布、搜索、详情、收藏、上下架 | 先覆盖数码/普通闲置 |
| 交易 | 卖家自出、平台验货两种模式 | 同一商品只能成交一次 |
| 支付 | 钱包支付、支付宝沙箱 | 必须防重复支付 |
| 物流 | mock 物流轨迹、确认收货 | 不接真实快递 |
| 验货 | 展示验货结果和报告 | 结果先由系统模拟 |
| 售后 | 退款申请、卖家审批、拒绝重提 | 退货物流先不做 |
| 后台 | 用户管理、提现审核、基础看板 | 不做复杂运营工具 |

## 3. 核心用例

| ID | 角色 | 用例 | 验收标准 |
| --- | --- | --- | --- |
| U1 | 买家 | 搜索并查看商品详情 | 可看价格、成色、卖家、交易模式 |
| U2 | 卖家 | 发布商品 | 必填项校验通过后商品可上架 |
| U3 | 买家 | 创建订单并支付 | 支付成功后订单状态正确推进 |
| U4 | 卖家 | 对自出订单发货 | 只有卖家本人可发货 |
| U5 | 买家 | 查看平台验货订单 | 订单页可看到验货报告和物流 |
| U6 | 买家 | 确认收货 | 未到可签收节点时不能确认 |
| U7 | 买家/卖家 | 发起与处理退款 | 买家可申请，卖家可同意或拒绝 |
| U8 | 管理员 | 管理用户与提现 | 普通用户不能进入后台 |

## 4. 竞品分析表

| 竞品 | 定位 | 强项 | 机会点 | 对 TrueUsed 的启发 |
| --- | --- | --- | --- | --- |
| 闲鱼 | 全品类个人二手交易 | 品类广、流量大、搜索和消息强 | 个人交易信任成本高 | 不拼全品类，先把验货和售后做扎实 |
| 转转 | 二手交易 + 官方验 | “官方验”心智清晰，适合 3C | 平台质检成本高 | 验货报告要放进订单主流程 |
| 爱回收 | 回收、以旧换新、严选二手 | 门店、上门、快递回收链路完整 | 更偏回收，不是自由交易 | 本期不做回收报价，聚焦买卖闭环 |

参考来源：[闲鱼官网](https://www.goofish.com/)、[转转官网](https://www.zhuanzhuan.com/)、[爱回收官网](https://www.aihuishou.com/)。

## 5. 技术选型矩阵

| 方案 | 内容 | 优点 | 风险 | 结论 |
| --- | --- | --- | --- | --- |
| A. 现有栈加固 | Vue 3 + Spring Boot 3 + Java 21 + MySQL + Redis + Docker | 已有代码多，交易状态好控制，适合答辩和交付 | 需要补权限、并发、支付幂等测试 | 推荐 |
| B. 云托管轻后端 | 前端 + Supabase/CloudBase 等托管服务 | 启动快，运维轻 | 订单、支付、权限边界容易绕复杂 | 不选 |
| C. 拆分服务 | 商品、订单、支付、消息拆成独立服务 | 后期扩展强 | 当前规模过重，调试成本高 | 后置 |

技术约束：

- API 返回统一结果结构。
- 登录、订单、地址、钱包、后台必须做权限校验。
- 异常交给全局处理，不能静默失败。
- Docker Compose 保留一键启动能力。

## 6. 验收指标

- 买家从浏览商品到支付、收货、评价的路径可跑通。
- 卖家从发布商品到发货、处理退款的路径可跑通。
- 平台验货订单能展示报告、物流和售后入口。
- 匿名用户不能创建订单、收藏、退款或进入后台。
- 重复支付或重复回调不会重复改订单或资金。

## 7. 后续优先级

1. P0：补齐权限边界、订单并发保护、支付幂等。
